# BACKTEST.md

Руководство по сбору данных и запуску бэктеста.

---

## Как это работает

Бэктест состоит из двух этапов:

1. **Сбор данных** — бот пишет снапшоты стакана и тиковые сделки в БД во время живой торговли
2. **Прогон** — скрипт читает сохранённые данные, прогоняет через стратегию без реальных ордеров и считает P&L

Симулятор повторяет логику реального бота: стоп-лосс, тейк-профит, тайм-аут, минимальное время удержания, cooldown, торговые часы.

---

## Шаг 1. Включить запись данных

В `.env` добавить:

```
RECORD_MARKET_DATA=true
```

Опционально — записывать не каждый снапшот стакана, а каждый N-й (для экономии места):

```
RECORD_ORDERBOOK_INTERVAL=2
```

> **Объём данных**: SBER генерирует ~150–300k снапшотов стакана в день (5–10 обновлений/сек за 8 часов торговли). При `RECORD_ORDERBOOK_INTERVAL=1` это ~50–80 МБ в день в MySQL. При значении `2` — вдвое меньше, но OFI будет считаться на более редких снапшотах.

Перезапустить бота — данные начнут накапливаться автоматически.

---

## Шаг 2. Проверить что данные есть

```bash
python3.11 backtest.py --ticker SBER --list-dates
```

Вывод:
```
Данные есть за: 2026-04-08, 2026-04-09
```

---

## Шаг 3. Запустить бэктест

### По одной дате

```bash
python3.11 backtest.py --ticker SBER --date 2026-04-08
```

### По диапазону дат

```bash
python3.11 backtest.py --ticker SBER --date-from 2026-04-07 --date-to 2026-04-09
```

### С переопределением параметров стратегии

Параметры берутся из `instruments.yaml`, но любой можно изменить без перезапуска бота:

```bash
python3.11 backtest.py --ticker SBER --date 2026-04-08 \
  --ofi-threshold 0.5 \
  --print-multiplier 8.0 \
  --stop-ticks 20 \
  --take-profit-ticks 80 \
  --min-ofi-confirmations 2
```

---

## Все параметры CLI

| Параметр | По умолчанию | Описание |
|---|---|---|
| `--ticker` | `SBER` | Тикер инструмента |
| `--date` | — | Дата бэктеста `YYYY-MM-DD` |
| `--date-from` | — | Начало периода `YYYY-MM-DD` |
| `--date-to` | — | Конец периода `YYYY-MM-DD` (включительно) |
| `--ofi-threshold` | из yaml | Порог OFI для входа (0–1) |
| `--print-multiplier` | из yaml | Мультипликатор принта |
| `--stop-ticks` | из yaml | Стоп-лосс в тиках |
| `--take-profit-ticks` | из yaml | Тейк-профит в тиках (0 = отключён) |
| `--min-ofi-confirmations` | из yaml | Подтверждений OFI для выхода |
| `--commission` | `0.0005` | Комиссия за одну сторону (0.05%) |
| `--list-dates` | — | Показать даты с данными и выйти |

## Подбор параметров

Типичный процесс:

1. Собрать данные за несколько дней (2–5 дней минимум)
2. Запустить бэктест с текущими параметрами — посмотреть на количество сделок и P&L
3. Если **сделок мало** — снизить `ofi_threshold` или `print_multiplier`
4. Если **много стопов** — увеличить `stop_ticks` или снизить `take_profit_ticks` для реалистичного соотношения
5. Если **много преждевременных выходов** по `ofi_reversed` — увеличить `min_ofi_confirmations`
6. Обновить `instruments.yaml` с лучшими параметрами и перезапустить бота

> **Важно**: бэктест на тех же данных, на которых подбираются параметры — это переобучение. Чем больше дней собрано, тем надёжнее результат.

## Таблицы БД

| Таблица | Что хранит |
|---|---|
| `market_orderbooks` | Снапшоты стакана: figi, bids (JSON), asks (JSON), recorded_at |
| `market_trade_ticks` | Тиковые сделки: figi, price, quantity, direction, recorded_at |

Индексы по `(figi, recorded_at)` на обеих таблицах — запросы по диапазону дат быстрые.

### Очистка старых данных

```sql
-- Удалить данные старше 30 дней
DELETE FROM market_orderbooks WHERE recorded_at < NOW() - INTERVAL 30 DAY;
DELETE FROM market_trade_ticks WHERE recorded_at < NOW() - INTERVAL 30 DAY;
```
